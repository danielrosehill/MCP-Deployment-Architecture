# v1 — 2026-04-26

Initial documented model. Two-tier MCP aggregation with a lightweight HTTP staging service for cross-machine file transfers.

## Aggregators

### MCP Aggregation 1 (LAN VM) — Default

The primary aggregator runs on a local virtual machine on the LAN. This is the default destination for all MCP servers unless there is a specific reason to run them locally.

Hosts connections to remote SaaS APIs (Google Workspace, Replicate, Pinecone, Meno) and infrastructure services running within the Docker network (e.g., PostgreSQL for conversation records).

The aggregation layer is MCP Jungle (formerly known as MetaMCP).

### MCP Aggregation 2 (Localhost) — Exception-Based

A second aggregator runs on the desktop machine, used only when one of the following conditions applies:

- The MCP **requires local device access** to function (e.g., Blender MCP, Revit MCP, or any tool that drives local desktop software).
- The MCP **does not support file transfer to a remote** and needs direct filesystem access.
- The MCP is **too cumbersome to configure remotely** (e.g., a transcription MCP that expects local audio hardware or file paths).

If none of these conditions apply, the MCP belongs on the LAN VM.

## Remote Access

Access to the LAN VM aggregator from outside the local network:

- **Cloudflare Tunnels** — stable, publicly-routable access without exposing ports.
- **Tailscale** — private mesh networking when on the move.

An **OpenClaw gateway** sits in front of the VM aggregator, handling routing for both local and remote clients.

## LAN Device Integration

Lightweight MCP servers run directly on local LAN devices:

- **Home Assistant** (home automation)
- **NAS** (network storage)
- **OPNsense** (firewall/router)
- **SBC Aggregator** — a dedicated single-board computer that aggregates signals from other SBCs (RPi 1, RPi 2, etc.)

These device-level MCPs feed into a **LAN Aggregator/Gateway**, which is then connected to MCP Aggregation 1 on the VM. Individual devices expose only their own tools; the aggregator composes them into a unified interface.

## File Transfer

A lightweight HTTP staging service runs alongside the MCP Jungle stack on the LAN VM. Any HTTP client (curl, browser, phone app) can:

1. `POST /upload` a file via standard multipart form data
2. Get back a `stagingPath` on the server's local filesystem
3. Pass that path to any MCP tool's `sourcePath` parameter

This bridges the binary-file-transfer gap in MCP (JSON-RPC has no binary channel) without requiring SSH/SCP from the client side.

## Design Rationale

- **Client portability** — centralising MCPs on a network-hosted VM means the tool surface is available to any AI client (desktop apps, CLI tools, remote agents) without per-client configuration.
- **Location flexibility** — Cloudflare + Tailscale make the same MCP servers reachable on the LAN or remotely, with no VPN juggling or port forwarding.
- **Mitigating tool bloat** — hierarchical aggregation keeps any single client's context window free from hundreds of irrelevant tools: device MCPs are narrow, the LAN aggregator composes them, the VM aggregator presents a curated surface, and the localhost aggregator handles only exceptions.
